# 통역사 학습 챗봇 서비스 비전

> 생성일: 2026-02-02
> 상태: Idea → DB 확장 완료 후 기획 시작

---

## 핵심 컨셉

통역사가 **실전 상황을 반복 연습**할 수 있는 AI 튜터 챗봇

## 사용자 시나리오

```
통역사: "3일 후에 삼성전자 베트남 공장 품질검수 통역 있어요"
챗봇:   → 제조 + 전자 + 품질관리 도메인 용어 조합
        → 해당 상황 시뮬레이션 반복 출제
        → 통역사 약점 파악 → 집중 훈련
```

## 서비스 흐름

```
1단계: 통역사가 "다가오는 통역 상황" 입력
2단계: 챗봇이 관련 도메인 용어 DB에서 상황 생성
3단계: 롤플레이 반복 훈련 (한→베, 베→한)
4단계: 통역사 응답 평가 + 약점 분석
5단계: 오답/실수 패턴 → 새로운 데이터로 축적
```

## 데이터 플라이휠 (핵심)

```
검증된 용어 DB (2만개)
      ↓
  챗봇 학습 서비스
      ↓
  통역사 사용 데이터 수집
  - 자주 틀리는 용어
  - 새로운 표현/용어
  - 상황별 난이도
  - 실제 통역 피드백
      ↓
  데이터 분석 → DB 보강
      ↓
  서비스 진화 (더 정확한 출제, 개인화)
      ↓
  (반복)
```

## 기술 아키텍처 (초안)

| 레이어 | 역할 | 기술 후보 |
|--------|------|----------|
| 용어 DB | Ground Truth | Supabase / JSON |
| RAG | 상황별 용어 검색 | Embedding + Vector DB |
| LLM | 대화 생성/평가 | Haiku급 (비용 최적) |
| 프론트 | 채팅 UI | Next.js / React |
| 분석 | 학습 데이터 수집 | 사용 로그 → 분석 파이프라인 |

## 경쟁 해자

1. **검증된 2만개 용어 DB** — 구축 비용 높음, 복제 어려움
2. **사용자 데이터 플라이휠** — 쓸수록 좋아짐, 후발주자 따라잡기 힘듦
3. **니치 특화** — 범용 GPT가 이길 수 없는 영역

## 선행 조건

- [ ] 20개 도메인 × 1,000개 = 2만개 용어 DB 완성
- [ ] 핵심 도메인 네이티브 검수 (의료, 법률, 건설)
- [ ] 서비스 기획 8단계 진행
- [ ] MVP 범위 정의

## 수집할 새 데이터 유형

| 데이터 | 활용 |
|--------|------|
| 오답 패턴 | commonMistakes 보강 |
| 사용자 표현 | 자연스러운 대안 표현 추가 |
| 난이도 체감 | 용어별 난이도 태깅 |
| 상황 조합 | 실제 통역 현장의 도메인 조합 패턴 |
| 피드백 | 용어 설명/예문 품질 개선 |
